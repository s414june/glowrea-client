# Feature: 書籤時間軸（Bookmarks）

## Goal

讓已登入使用者能查看自己儲存的書籤貼文列表，作為「稍後再看」的私人收藏入口。

書籤與「喜歡」的定位不同：書籤偏向內容收藏（之後再看），喜歡偏向情緒表達；兩者均不公開顯示。

---

## Scope

### In Scope

- 顯示書籤貼文列表（`GET /api/v1/bookmarks`）
- 首次載入資料
- 顯示 loading / empty / error state
- 支援載入更多（infinite scroll）
- 需登入才可存取，未登入導向 `/login`
- 複用 `TimelineList`、`StatusItem`、`LoadingSkeleton` 等現有元件

### Out of Scope

- 新增 / 移除書籤的互動操作
- 書籤分類或分組
- 批次管理（全部刪除、匯出）
- 搜尋書籤

---

## User Story

作為已登入使用者，我希望能查看我儲存的書籤貼文，以便回顧之前想保留的內容。

---

## User Flow

1. 使用者進入 `/bookmarks`
2. 頁面顯示 loading skeleton
3. API 回傳後，顯示書籤貼文列表（依加入時間由新到舊）
4. 列表為空時顯示 empty state
5. 向下捲動到底部時，自動載入更多
6. API 發生錯誤時，顯示 error state 與 retry 按鈕

---

## Route

### Path

- `/bookmarks`

### Route Guard

- 需登入；使用既有 `auth.global.ts` middleware

### Route Responsibility

- 頁面只負責初始化與渲染
- 不在 route 層處理資料轉換邏輯

---

## UI Requirements

### 頁面標題區

- 顯示「書籤」標題（小型 section header）
- 不需返回按鈕

### Loading State

- 使用現有 `LoadingSkeleton` 元件

### Empty State

- 說明文字：「還沒有儲存任何書籤」

### Error State

- 顯示錯誤訊息 + Retry 按鈕

### 列表

- 複用 `TimelineList` 元件
- Infinite scroll sentinel（`IntersectionObserver`），與 home timeline 一致

---

## 資料流

### API Endpoint

`GET /api/bookmarks`

query params：
- `maxId?: string`

### 對應 Mastodon API

```
GET /api/v1/bookmarks?limit=20[&max_id={maxId}]
```

需要 Authorization header。

### 回傳格式

複用現有型別：

```ts
TimelinePageResponse = {
  items: TimelineStatus[]
  nextMaxId: string | null
}
```

### 分頁

Link header `rel="next"` → 解析 `max_id` 參數，與 home timeline 一致。

## Server Route 規格

### `GET /api/bookmarks`

- 呼叫 `requireRequestCredentials` 取得 token
- 呼叫 `GET /api/v1/bookmarks?limit=20[&max_id=...]`
- 帶 Authorization header
- 回傳 `TimelinePageResponse`
- 解析 Link header 取得 `nextMaxId`

---

## 錯誤處理

| 情境 | 行為 |
|------|------|
| Mastodon API 回傳非 2xx | 回傳對應 HTTP 狀態碼給前端 |
| 未登入 | 401，middleware 導向 `/login` |
| 網路錯誤 | 前端顯示 error state + retry |
| 列表為空 | 顯示 empty state，不視為錯誤 |
